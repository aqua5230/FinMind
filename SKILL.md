# Skill: FinMind 專案協作流程

## 一鍵執行入口（Entry Command）

每一步統一使用以下流程：

**1. Claude 輸出「最終執行版任務單」**

**2. 使用者執行：**
```bash
codex exec --full-auto --skip-git-repo-check "[任務單]"
```

**3. 執行完成後自動進入驗收：**
```bash
# (1) Gemini 正確性審查
gemini -p "你是嚴格的程式碼審查員。禁止修改任何程式碼，只做檢查。
[貼入完成標準]
只回報：PASS 或 問題清單（條列）。" --model gemini-2.5-pro --yolo

# (2) Codex 範圍審查
codex exec --full-auto --skip-git-repo-check "你是範圍審查員。禁止修改任何程式碼，只做檢查。
[貼入範圍檢查清單]
逐條回報 PASS 或問題。總結：PASS 或 FAIL+問題清單"
```

**4. 根據結果：**
- PASS → 進下一步
- FAIL → 使用「任務重試模板」修復，重跑驗收

---

## 角色分工

| 角色 | 工具 | 職責 |
|------|------|------|
| Claude | — | 架構師：拆解任務、定義邊界、產出任務單、整合驗收結果。 |
| Codex | `codex exec --full-auto --skip-git-repo-check "..."` | 執行者：修改前後端程式碼、補文件、建立必要檔案。 |
| Gemini | `gemini -p "..." --model gemini-2.5-pro --yolo` | 驗收者：確認 API、部署行為、資料格式與需求符合。 |
| Codex（第二個） | 同上 | 範圍審查：確認只修改指定檔案，沒有越界變更。 |

## 任務單格式（每步固定）

```
1. 目標
   一句話說明這步做什麼，不包含下一步。

2. 要建立或修改的檔案
   - 列出每個檔案
   - 精確到每一行的改法（原本 → 改成）
   - 整段刪除或新增要列出完整內容

3. 不能碰的檔案
   明確列舉所有禁止修改的檔案與資料夾

4. 完成標準
   - 可用 grep、rg、npm、python3、curl 驗證的具體條件
   - 不寫「看起來正確」，只寫可機器驗證的條件
```

**原則：**
- 一步只做一件事
- 不包含下一步的內容
- 完成標準必須可機器驗證

## 執行流程

```
1. Claude 先讀專案現況與限制
2. Claude 輸出第 N 步任務單
3. 使用者確認任務單（可提出修正）
4. Claude 確認修正點後輸出最終執行版任務單
5. Codex 執行
6. 雙線驗收（Parallel）：
   - Gemini → 正確性審查
   - Codex  → 範圍審查
7. 兩份報告都 PASS 才放行，進下一步
8. 任一 FAIL → 彙整問題，重新派 Codex 修（算入重試次數）
9. 重試上限 3 次，超過回報使用者
```

---

## 驗收重點（FinMind）

- 前端 `frontend/` 的 API 請求是否使用相對路徑或正確的部署 URL
- Next.js 設定是否可在 Railway / 本機一致運作
- 後端 `/api/health`、`/api/price/:stockId` 是否回傳正常
- 股票資料圖表與股票名稱查詢是否仍可載入
- 新增文件是否反映目前專案分工與流程

---

## 驗收指令模板

### Gemini 正確性審查
```bash
gemini -p "你是嚴格的程式碼審查員。禁止修改任何程式碼，只做檢查。

## 任務單完成標準
[貼入本步的完成標準]

請讀取相關檔案，必要時呼叫 API 驗證。只回報：PASS 或 問題清單（條列）。" \
  --model gemini-2.5-pro --yolo
```

### Codex 範圍審查
```bash
codex exec --full-auto --skip-git-repo-check "你是範圍審查員。禁止修改任何程式碼，只做檢查。

## 檢查清單
1. 禁止修改的檔案時間戳是否早於本步修改的檔案（確認未被碰）
2. 新建/修改的檔案是否只有任務單列出的那些
3. 本步要求的設定、路由、文件內容是否存在

逐條回報 PASS 或問題。總結：PASS 或 FAIL+問題清單"
```

## Fallback 機制

| 狀況 | 處理方式 |
|------|---------|
| Gemini-2.5-pro 429 quota | 改用 Codex 範圍審查替代，仍視為有效驗收 |
| gemini-2.0-flash 404 | 不使用此模型，改用 Codex |
| Gemini 誤判（如時間戳理解錯誤） | 架構師說明原因，記錄備注，放行 |
| Codex 執行失敗 | 診斷原因（環境/需求/邏輯），縮小任務重新派，最多 3 次 |
| 第 3 次仍失敗 | 回報使用者，不再自動重試 |

---

## 任務執行限制（Execution Lock）

- 嚴禁修改未列在任務單中的檔案
- 嚴禁順手重構、格式化整個專案、改命名風格
- 嚴禁調整 `.env`、token、憑證與部署秘密
- 若發現額外問題但不在本步範圍，只記錄不修改

---

## 任務重試指令模板

```
請修正以下問題（只修這些，不得修改其他程式碼）：

[條列問題，每條包含：檔案、行號、問題描述、期望結果]

限制：
- 只允許修改與問題直接相關的程式碼
- 不得新增或刪除其他邏輯
- 不得影響已通過的部分
```

重試後重走完整驗收流程（Gemini + Codex），算入本步重試次數。

---

## Bug 處理規則

- 驗收中發現與本步**無關**的舊 bug → 記錄，不在本步處理
- 驗收中發現本步**任務單遺漏**的問題 → 屬於本步範圍，算第一次重試
- 修補遺漏只修那一處，不擴大範圍
- 每步重試上限獨立計算（各 3 次）

---

## 常見任務類型（FinMind）

### 1. 前端 API 串接
- 修改 `frontend/lib/` 或 `frontend/hooks/` 的請求邏輯
- 必要時更新 `frontend/next.config.*`
- 完成標準要包含實際路徑檢查

### 2. 後端 API / 報表邏輯
- 修改 `stock_report/` 或 `main.py`
- 驗收需確認路由、資料格式、例外處理

### 3. 部署與文件補建
- 只補必要設定與文件
- 不把部署細節分散在多份重複文件

---

## 最終驗收（Final Gate）

**1. 檢查本步修改檔案**
```bash
git diff --name-only
```

**2. 檢查前端設定與 API 路徑**
```bash
rg -n 'rewrites|/api/:path\\*|API_URL' frontend
```

**3. 檢查文件存在**
```bash
test -f SKILL.md && test -f TASK_LOG.md && echo ok
```

**4. 需要時執行專案檢查**
```bash
cd frontend && npm run lint
```

全部通過才算結案。任一失敗，定位到對應步驟補修後重跑。
